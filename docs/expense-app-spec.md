# 経費申請アプリ 仕様書

最終更新: 2026-07-17

## 1. 目的

経費の申請から確認・承認までをブラウザ上で完結させ、申請データを
**Google スプレッドシートにデータベース化**、**領収書画像を Google ドライブに保存**する。
ログインと権限管理（一般／管理者）を備え、蓄積データを実績管理・分析ツールから
読み込んで連携・分析できる状態にする。

## 2. システム構成

```
[利用者のブラウザ]                        [Google クラウド]
  経費申請アプリ (静的サイト)               ┌───────────────────────────┐
   - ログイン / セッション管理             │ Apps Script Web App       │
   - 画像OCR (Tesseract.js / 端末内)  ─┐   │  認証・権限チェック        │
   - 申請フォーム                      │   │  doPost: auth/create/     │
   - 個人 / 管理者ダッシュボード        ├──▶│   update/delete/users     │
   - localStorage キャッシュ           │   │  doGet : JSON（権限別）    │
                                       │   └───────┬───────────┬───────┘
                                       │           ▼           ▼
                                       │   スプレッドシート   Google ドライブ
                                       │   expenses / users  (領収書画像)
                                       │           │
[分析・実績管理ツール] ◀────────────────┘           │
   Looker Studio / BIツール / スプレッドシート集計 / JSON API / CSV
```

- フロントエンド: バニラ JS（`expense-app/`）。GitHub Pages で公開。レスポンシブ対応。
- バックエンド: Google Apps Script Web App（`apps-script/Code.gs`）。サーバー・GCP設定不要。
- データストア（1つのスプレッドシート内）:
  - `expenses` シート = 経費データ（1申請1行）
  - `users` シート = ユーザーアカウント（パスワードはソルト付き SHA-256 ハッシュ）
  - Google ドライブ「経費領収書」フォルダ = 領収書画像（行に URL を保持）

### 2.1 動作モード

| モード | 条件 | 認証 | 保存先 |
| --- | --- | --- | --- |
| クラウド | ⚙️でWebアプリURLを設定 | ログイン必須 | スプレッドシート＋ドライブ（正本） |
| ローカル（試用） | 未設定 | なし（氏名入力のみ） | localStorage のみ |

クラウドモードで通信失敗した申請は端末内の再送信キューに退避し、「再同期」で反映する。

## 3. 認証・権限

### 3.1 認証方式
- ユーザーID＋パスワード。パスワードは `users` シートにソルト付き SHA-256 で保存。
- ログイン成功で **HMAC-SHA256 署名付きセッショントークン**（有効期限12時間）を発行。
  以降のリクエストはトークンを必須とし、サーバー側で毎回検証する。
- 初回接続時（`users` が空）は「初期設定」画面で最初の管理者を作成する
  （それまでは従来互換のオープンモード）。

### 3.2 権限（ロール）

| 操作 | 一般（user） | 管理者（admin） |
| --- | --- | --- |
| 経費の申請 | ✅（申請者名・事業部はセッションからサーバーが強制） | ✅ |
| 申請の閲覧 | 自分の申請のみ | 全申請 |
| 承認 / 却下 / 差戻（フローモード時のみ） | ❌ | ✅（承認者名もサーバーが強制） |
| 申請の削除 | 自分の申請（状態問わず・ミス時の自己取消） | 全件（ミス・重複申請の削除） ※いずれもシート行＋ドライブ画像を連動削除 |
| ユーザー管理（追加・無効化・PW再設定・事業部設定） | ❌ | ✅（自分自身の無効化・降格は不可） |
| パスワード変更（自分） | ✅ | ✅ |

### 3.3 承認モード

- **自動承認モード（既定）**: 申請は保存と同時に `approved` になる（承認者=「自動承認」）。
  承認フローは省略し、誤り・重複は管理者の削除で対応する。
- **フローモード**: スクリプトプロパティ `AUTO_APPROVE` を `"false"` に設定すると、
  従来の 申請中 → 承認/却下/差戻 フローが有効になる（将来のスケール時に復活可能）。
  アプリUIはモードに応じて操作ボタンを自動で出し分ける。

## 4. 機能要件

### 4.1 画像解析（レシート読み取り・2段構え）
- レシート／領収書画像をアップロード（クリック選択・ドラッグ＆ドロップ）。**画像は任意**。
- **3段フォールバック**: AI解析 → 端末内OCR → 手入力。
- **①AI解析（高精度）**: GAS に AIキー設定時、画像解析＋構造化出力で
  **金額・日付・店名・科目・摘要** を抽出しフォームへ自動入力。プロバイダは設定で選択:
  - **Gemini（推奨・無料枠可）**: `GEMINI_API_KEY`。モデルは `GEMINI_MODEL`（既定 `gemini-2.5-flash`）。
    無料枠は入力/出力が学習に利用される点に留意。
  - **Claude**: `ANTHROPIC_API_KEY`。モデルは `OCR_MODEL`（既定 `claude-opus-4-8`）。学習利用なし。
  - 両方設定時は Gemini 優先（`OCR_PROVIDER` で切替）。
- **②端末内OCR（フォールバック）**: AI未設定時・失敗時は Tesseract.js。
  前処理（拡大→グレースケール→大津の二値化）と単一ブロック解析で精度を強化。
  金額抽出は「合計/総額/請求」優先、「お預り/お釣り/現金」等を除外。
- **③手入力**: いずれも読み取れないとき。抽出結果はフォームへ自動入力され手修正可能。

### 4.2 経費申請
- 入力: 日付（必須）、科目（必須）、事業部、支払先、金額（必須・1円以上）、摘要。
- **領収書画像は任意**（画像なしでも申請可能）。
- **事業部は申請ごとにプルダウンで選択可能**（事業部マスタから選択）。
  未選択の場合は自分のプロフィールの事業部が自動で入る。
- 科目: 交通費／交際費／会議費／消耗品費／通信費／宿泊費／その他。

### 4.2.1 事業部マスタ
- 事業部は `departments` シートで管理し、申請フォーム・ユーザー登録のプルダウンに反映。
- 初期値: BAR / 人材 / 運送 / 本部 / ARTGRAGE / クリニック / GoonerHouse。
- 管理者ダッシュボードの「事業部の管理」から追加・削除が可能。
  削除しても過去の申請データの事業部名は保持される（選択肢から外れるだけ）。

### 4.2.2 領収書の保管（Google ドライブ）
- フォルダ体系: `経費領収書/<事業部>/<yyyy-MM>/`。事業部未設定の申請は「未設定」フォルダへ。
- **事業部フォルダ**: 事業部の追加と同時に自動作成。事業部を削除しても
  **フォルダと保存済み画像は残す**。
- **月フォルダ**: 申請保存時に経費の日付の月フォルダを自動作成（抜け漏れなし）。
  加えて `setupMonthlyFolderTrigger()` を一度実行すると、**毎月末（28日）に
  翌月分のフォルダを全事業部へ事前生成**するトリガーが登録される。
- **ファイル名規則**: `日付_申請者_採番.jpg`（例: `2026-07-15_山田太郎_001.jpg`）。
  採番は同じ日付・同じ申請者の中で 001 からの連番。申請の削除時は画像も
  ゴミ箱へ連動（従来どおり）。

### 4.3 個人ダッシュボード
- **月次表示**: 既定は当月。◀▶で月移動、月指定、「全期間」切替が可能。
- 自分の申請一覧・集計（申請件数、申請中、承認済み金額、申請中金額）。
- 申請中の取消、領収書リンク、却下理由の確認、パスワード変更。

### 4.4 管理者ダッシュボード
- **月次表示**: 既定は当月。◀▶で月移動、月指定、「全期間」切替が可能。
- **グループ別集計グラフ**: 区分（科目）／事業部／スタッフを切り替えて
  承認済み金額を可視化。
- **月次推移グラフ**: 直近6ヶ月の承認済み金額。
- 全申請の一覧・検索（申請者・店名・事業部）・状態フィルタ・
  **ソート**（日付／金額／スタッフ／事業部／区分）・CSV書き出し。
- 申請の**削除**（ミス・重複対応。シート行＋ドライブ画像を連動削除）。
  フローモード時は承認 / 却下（理由入力）/ 差戻も表示。
- ユーザー管理: 一覧・追加（ID/表示名/事業部/初期PW/権限）・無効化/有効化・
  PW再設定・事業部変更（インラインのプルダウン）。
- 事業部マスタ管理: 事業部の追加・削除（プルダウンの選択肢に反映）。

### 4.5 UI
- レスポンシブ対応（モバイル〜デスクトップ）。ライトテーマ。
- デザインは今後さらに改善予定（機能を優先して構築）。

## 5. データモデル

`expenses`: `id, createdAt, applicant, date, category, vendor, amount, description,
status, reviewedAt, reviewer, reviewComment, imageUrl, imageFileId, applicantId,
department`

`users`: `username, displayName, passwordHash, salt, role, active, createdAt,
department`

`departments`: `name`（事業部マスタ。初期値 BAR / 人材 / 運送 / 本部 /
ARTGRAGE / クリニック / GoonerHouse）

- `status`: `pending` / `approved` / `rejected`。`role`: `user` / `admin`。

## 6. API（POST は text/plain の JSON。認証系以外は token 必須）

| action | 権限 | 内容 |
| --- | --- | --- |
| `status` | 不要 | 認証が有効か（初期設定済みか）を返す |
| `setup` | 不要（初回のみ） | 最初の管理者を作成しトークン発行 |
| `login` | 不要 | 認証してトークン発行 |
| `me` | user | セッション検証・ユーザー情報 |
| `changePassword` | user | 自分のパスワード変更 |
| `create` | user | 申請作成（画像base64があればDrive保存） |
| `update` | admin | 承認・却下・差戻 |
| `delete` | user/admin | 取消（userは自分の申請中のみ） |
| `listUsers` / `upsertUser` | admin | ユーザー管理 |
| `listDepartments` | user | 事業部マスタ取得（プルダウン更新用） |
| `addDepartment` / `deleteDepartment` | admin | 事業部マスタの追加・削除 |
| `analyzeReceipt` | user | AIレシート解析（`ANTHROPIC_API_KEY` 設定時のみ） |
| GET `?token=` | user/admin | JSON取得（userは自分の分のみ） |

分析ツール向け: スクリプトプロパティ `SHARED_TOKEN` を設定すると、
GET `?token=<SHARED_TOKEN>` で全件を読み取り専用取得できる（Looker Studio等の定期取得用）。

主なスクリプトプロパティ: `AUTO_APPROVE`（`"false"`で承認フロー有効。既定は自動承認）、
`SPREADSHEET_ID` / `DRIVE_FOLDER_ID` / `AUTH_SECRET`（自動保存）、`SHARED_TOKEN`。

保存先は共有ドライブへ移行可能（ファイルIDは移動しても不変のため設定変更不要。
GAS実行アカウントを共有ドライブの「コンテンツ管理者」以上にすること）。

## 7. セキュリティ / 制約

- パスワードは平文保存しない（ソルト付きSHA-256）。トークンはHMAC署名・12時間で失効。
- 申請者名・承認者名はサーバー側でセッションから強制し、なりすましを防止。
- スプレッドシート・ドライブ自体の閲覧権限は Google 側の共有設定に従う
  （管理者=シート所有者のみが直接閲覧可能な構成を推奨）。
- ローカルモードは試用向けで認証なし。データは端末・ブラウザ間で共有されない。
- OCR は端末内処理。申請時に画像・内容は自組織の Google 環境へ送信される。
